# Mode-specific Detection Training

This workflow keeps model training separated by runtime mode. The same
`segment_filter` / `edge_trim_advisor` architecture can be reused, but each
mode should train from candidates generated by that mode and pass its own
regression gate before being installed.

## Profiles

All three modes now pass an explicit `smoothingProfile` before segment filter
finalization. The initial rule values intentionally match the previous shared
heuristic; future tuning must change only the target profile and then pass its
own A/B gate.

Training summaries may carry `ignoreRanges` / `evaluationIgnoreRanges` either
at the top level or under `params`. `train_segment_filter.py` skips candidate
examples that overlap those ranges, so manually omitted partial or ambiguous
performances are not mined as negatives. Keep this enabled unless the ignored
range was added by mistake.

### `offline-final`

Purpose: Offline Workbench final output with full-audio context.

Runtime notes:

- Long audio is decoded and analyzed in overlapped AED chunks to avoid OOM.
- Smoothing and profile model finalization run only after the full
  `offline-final` frame cache is available.
- Waveform/spectrogram rendering is separate from AED analysis. For long audio
  it runs later through visual-only workers and stitches chunk visuals; visual
  failures must not invalidate AED results.

Training candidates:

- `evaluate_smoothing_batch.py` summaries.
- Full AED frames from the offline diagnostic cache.
- Full manual annotations.

Command shape:

```powershell
python tools/evaluate_smoothing_batch.py `
  --out-dir training_runs/smoothing_eval_offline_final

python tools/train_segment_filter.py `
  --training-profile offline-final `
  --batch-dir training_runs/smoothing_eval_offline_final `
  --out-dir training_runs/segment_filter_offline_final `
  --write-profile-assets

python tools/evaluate_smoothing_batch.py `
  --out-dir training_runs/smoothing_eval_offline_final_candidate `
  --segment-filter `
  --segment-filter-model-dir training_runs/segment_filter_offline_final `
  --segment-filter-profile offline-final `
  --require-profile-assets
```

Gate: aggregate F1 and mean F1 must not drop; severe outliers should decrease.
Use `python tools/run_offline_profile_replacement_gate.py` for the final
replacement check. It compares candidate `offline-final` assets against the
currently shipped profile assets and fails if aggregate F1, mean F1, or severe
outlier count regress. Candidate assets must also pass
`validate_profile_assets.py --profile offline-final --require-post-end-guard-metadata`.

### `live-pcm30`

Purpose: Live 30-minute PCM rollover finalization. This is the default live
path for higher fidelity final output.

Training candidates should come from PCM no-filter live simulation so the
filter learns the exact candidates produced by live rollover, not offline
candidates.

```powershell
node tools/live/run_live_pcm_ab_samples.mjs `
  --samples tools/samples/live/live_pcm_full_regression_samples.example.json `
  --variants pcm-no-filter `
  --include-frames `
  --out-dir training_runs/live_pcm30_training_candidates `
  --force

python tools/train_segment_filter.py `
  --training-profile live-pcm30 `
  --batch-dir training_runs/live_pcm30_training_candidates `
  --summary-glob "*_pcm_filter_off.json" `
  --out-dir training_runs/segment_filter_live_pcm30 `
  --write-profile-assets

node tools/live/run_live_pcm_ab_samples.mjs `
  --samples tools/samples/live/live_pcm_full_regression_samples.example.json `
  --variants pcm-current,pcm-candidate-model,pcm-no-filter `
  --candidate-segment-filter-model-dir training_runs/segment_filter_live_pcm30 `
  --out-dir training_runs/live_pcm30_candidate_ab `
  --jobs 3 `
  --fail-on-regression `
  --require-profile-assets
```

Gate: candidate must beat or match `pcm-current` on the primary, extended,
stop-checkpoint, snapshot-unavailable, and no-song suites before install.
Use `tools/live/run_live_profile_replacement_gate.mjs --profile live-pcm30`
for the final replacement check so sampled 3-video A/B and all required suites
run with `--require-profile-assets`. The gate validates candidate metadata with
`--require-post-end-guard-metadata` before running A/B.

### `live-realtime-aed60`

Purpose: Realtime 60-second AED cache path. This mode has less future context
than PCM finalization, so it should not inherit aggressive offline trims by
default.

```powershell
node tools/live/run_live_pcm_ab_samples.mjs `
  --samples tools/samples/live/live_pcm_full_regression_samples.example.json `
  --variants aed60-no-filter `
  --include-frames `
  --out-dir training_runs/live_aed60_training_candidates `
  --force

python tools/train_segment_filter.py `
  --training-profile live-realtime-aed60 `
  --batch-dir training_runs/live_aed60_training_candidates `
  --summary-glob "*_aed60_filter_off.json" `
  --out-dir training_runs/segment_filter_live_aed60 `
  --write-profile-assets

node tools/live/run_live_pcm_ab_samples.mjs `
  --samples tools/samples/live/live_pcm_full_regression_samples.example.json `
  --variants aed60-current,aed60-candidate-model,aed60-no-filter `
  --candidate-segment-filter-model-dir training_runs/segment_filter_live_aed60 `
  --out-dir training_runs/live_aed60_candidate_ab `
  --jobs 3 `
  --fail-on-regression `
  --require-profile-assets
```

Gate: candidate must not lower mean F1 versus `aed60-current`, and no-song
false positives must not increase. Realtime provisional output should remain
heuristic-first; model correction belongs to finalization.
Use `tools/live/run_live_profile_replacement_gate.mjs --profile live-realtime-aed60`
for the final replacement check. The gate validates candidate metadata with
`--require-post-end-guard-metadata` before running A/B.

## Runtime Policy

- Do not install mode-specific assets into `models/fireredvad/aed` until their
  profile-specific A/B gate passes.
- Use `tools/install_profile_model_assets.py` for installation after a gate
  passes. It revalidates candidate metadata, backs up existing profile files,
  and copies only profile-specific assets.
- Profile-specific asset names are supported:
  - `segment_filter_offline_final.*` / `edge_trim_advisor_offline_final.*`
  - `segment_filter_live_pcm30.*` / `edge_trim_advisor_live_pcm30.*`
  - `segment_filter_live_aed60.*` / `edge_trim_advisor_live_aed60.*`
- `train_segment_filter.py --install-mode profile` installs only the
  profile-specific filenames. `--install-mode both` also refreshes the legacy
  `segment_filter.*` fallback filenames.
- Extension runtime selection now requires the profile file for the active mode.
  If a profile asset is missing, that profile model is disabled and finalization
  falls back to heuristic smoothing instead of using another mode's model.
- The current release assets include pinned offline and live profile copies:
  `segment_filter_offline_final.*`, `edge_trim_advisor_offline_final.*`,
  `segment_filter_live_pcm30.*`, `edge_trim_advisor_live_pcm30.*`,
  `segment_filter_live_aed60.*`, and `edge_trim_advisor_live_aed60.*`.
  These are copies of the verified finalization model, with profile metadata
  added, so future mode-specific asset replacement does not silently alter
  another mode.
- Live profile segment filter metadata pins the accepted
  `liveFinalKeepThreshold` per asset. This prevents runtime constant changes
  from silently altering Live output. The accepted `live-pcm30` and
  `live-realtime-aed60` profiles both use `liveFinalKeepThreshold=0.5`;
  older or un-replaced profiles may still use a different pinned value.
- Live-profile `segment_filter.meta.json` may declare `liveFinalKeepThreshold`.
  The runtime uses this value only when present; older shipped models keep the
  conservative live final threshold floor.
- Live-profile `edge_trim_advisor.meta.json` may declare
  `disableLiveEdgeTrim=true`. Training sets this automatically when validation
  edge MAE is too high, because an unreliable edge advisor is worse than using
  no edge trim.
- `enableLiveEndTrimEvidenceGuard=true` should be enabled for live-profile edge
  advisor assets. After the corrected `video_015` annotation, primary and full
  PCM A/B both passed with this guard enabled; it prevents large model end trims
  when the proposed trimmed tail still has strong song evidence.
- Do not apply edge trim to discontinuity/snapshot finalization. A rejected
  experiment enabling discontinuity edge trim dropped snapshot mean F1 from
  `0.9636` to `0.9269`, so discontinuity sessions should keep edge trim off.
- Keep `minSegmentDurationSec` as a user option. Training and evaluation may
  pass the active value through, but should not change its user-facing behavior.
- If separate shipped assets are needed later, use distinct filenames such as
  `segment_filter_live_pcm30.onnx` only after the runtime selector and
  missing-profile behavior are tested.
- `tools/live/run_live_pcm_ab_samples.mjs --jobs N` runs sample/variant tasks in parallel.
  Use `--jobs 2` or `--jobs 3` for full PCM regression on a desktop machine;
  higher values can saturate disk/CPU because each task launches its own Node,
  FFmpeg, and ORT runtime.
- Use `--require-profile-assets` when validating profile-specific live assets.
  This makes `tools/live/simulate_live_pcm_detection.mjs` fail if `live-pcm30` or
  `live-realtime-aed60` falls back to default `segment_filter.*` or
  `edge_trim_advisor.*` files. The A/B runner records requested/loaded profiles
  in both JSON and CSV summaries.
- Run `python tools/validate_training_annotations.py` after editing
  `tools/annotations_example.csv` or `tools/data/manual/video_*_manual.txt`.
  It fails when CSV and manual files disagree, which prevents another
  `video_015`-style annotation pollution from entering retraining.
- Manual post-end negative mining now has a conservative song-evidence guard.
  If the `manual.end + 0.5s ~ +14s` window still contains temporal song and
  singing evidence without a clear speech/low-energy reset, it is skipped as a
  likely annotation-tail issue rather than forced into the negative class.
- `evaluate_smoothing_batch.py --jobs N` evaluates different offline samples
  in separate worker processes. Use it when the batch appears stuck in one
  long sample. Add `--command-timeout-sec <sec>` if an external diagnose,
  smoothing, or live-simulation command needs a hard cap.
